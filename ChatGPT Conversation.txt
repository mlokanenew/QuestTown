For this game, I would build it in Godot 4 with GDScript, and use Python alongside it for tooling, not for the runtime game itself.

Why that’s the best fit for your brief:

You want something a coding assistant can drive heavily, with as much of the project as possible living in normal text files rather than being trapped in opaque editor state. Godot is unusually strong here: GDScript is indentation-based and deliberately Python-like, but tightly integrated with the engine; Godot’s .tscn scene files are human-readable text; .tres resource files are version-control-friendly text; and Godot supports command-line running, debugging and headless export, which is excellent for assistant-driven workflows and CI. 

It also matches your game shape better than pure Python. Your design needs a real 3D engine, pathfinding/navigation for lots of town residents and adventurers, and a sane asset pipeline from generated/edited 3D assets into the game. Godot has built-in 3D navigation and pathfinding APIs, and it can import Blender scenes directly or work through glTF, which fits well with an LLM-assisted 3D asset pipeline. 

I would not start this in Unreal for your use case, and I would be cautious about Unity. Unreal’s workflow leans heavily on Blueprint visual scripting, which is powerful but pushes more game logic into node graphs that are less pleasant for a coding assistant to author and maintain. Unity has a strong C# stack, but its own docs and training material still assume substantial editor customisation, inspectors and editor windows, which tends to pull work back into the editor rather than keeping it in plain code and data. 

On the Python question specifically: I do not think “runtime game in Python” is the best choice here, even though you know Python well. Godot’s docs are quite explicit that they chose not to use Python directly because earlier Python/Lua integrations had serious limitations, including threading issues. That lines up with the practical reality: for a living 3D town sim with lots of agents, you want an engine whose main gameplay language is built for that engine. 

So my recommendation is:

Engine: Godot 4.x
Main gameplay language: GDScript
Secondary language: Python 3.12+ for offline tools
3D pipeline: Blender + glTF/.blend import into Godot
Data model: Godot Resources (.tres) for careers, buildings, quests, guilds, enemies, regions
Source control: Git from day one

That gives you the best split:

GDScript for the actual game.

Python for assistant-friendly generators and utilities:

career tree generators

quest content generators

economy/balance simulators

save-file analysers

batch asset-processing scripts

content import/export tools

The big design decision I’d make early is this:

Keep the game simulation mostly data-driven and code-driven, not editor-driven.

In practice that means:

careers are data files

buildings are data files

quest templates are data files

factions/guilds are data files

map chunks and prefabs are scenes

game rules live in scripts

UI pulls from data, not hardcoded branches everywhere

That is the sort of structure a coding assistant can keep extending safely.

For your particular game, I’d also avoid trying to make it “full Manor Lords 3D”. I would aim for:

a stylised 3D isometric / high-angle camera

fairly simple placement/building rules

visually rich but mechanically clear streets, districts, guild halls, inns, temples and markets

heroes as named simulated units with inventory, wounds, loyalties, career progress and relationships

the depth living in the sim and narrative systems, not in cutting-edge rendering

That choice matters because it keeps the project in the zone where an assistant can do a huge amount of the work.

There is one credible “all Python” fallback: Panda3D, with Ursina as a friendlier wrapper for fast prototyping. Panda3D is a real open-source 3D engine with Python bindings, and Ursina uses Python throughout. That said, I would only choose that route if your absolute top priority is “everything in Python no matter what”, because you will end up building more of the game-framework layer yourself. 

So the decision in plain English is:

Best overall choice: Godot + GDScript + Python tools

Best if you insist on stronger typing from day one: Godot + C#

Best if you insist on pure Python runtime: Panda3D/Ursina

Not my recommendation for assistant-first development: Unreal

Viable but more editor-heavy than you want: Unity

I would start with Godot + typed GDScript, not C#. Godot itself recommends GDScript as the easiest starting point with the tightest integration, while C# support is there when you need it.

Thought for 1m 21s
Yes — and I’d set this up from day one. The right pattern is not “let the LLM pretend to click buttons in the UI”. It is: make the game simulation callable as a headless service, and keep rendering as an optional layer on top. Godot supports headless execution with --headless, supports custom command-line args via -- plus OS.get_cmdline_user_args(), and supports dedicated-server style exports with a dedicated_server feature tag that can force headless mode and strip visual resources from the export. 

For your MVP, I’d ask the coding assistant to build the project around three modes:

Playable mode: normal camera, UI, mouse placement.

Headless sim mode: no rendering, no UI, accepts structured commands.

Scenario test mode: runs scripted scenarios, emits JSON results, exits with pass/fail. Godot can run headless from the command line, and --quit / --quit-after are useful for automation and CI-style runs. 

The most important architectural instruction for the assistant is this:

Split the project into simulation, presentation, and control.

simulation owns world state, taverns, heroes, spawning, path targets, economy stubs

presentation owns camera, meshes, animations, UI, selection outlines

control owns command parsing, scenario runner, and optional socket server

That split is mostly design advice rather than a Godot feature, but it matches Godot’s strengths because scenes and nodes are flexible, while headless mode disables rendering/window functions, so anything core to the game should not depend on visuals being present. 

The second instruction is:

Keep game logic out of _process() wherever possible. Put simulation in fixed-step logic.

Godot’s _process() runs as fast as rendering allows, while _physics_process() runs on a fixed tick rate, 60 Hz by default. For a local LLM to test the game repeatably, you want the sim to advance in fixed ticks, with a seeded RNG, rather than depending on framerate. 

So I would tell the assistant to implement a structure like this:

SimulationRoot.gd

TownState.gd

HeroSystem.gd

BuildingSystem.gd

SpawnSystem.gd

CommandServer.gd

ScenarioRunner.gd

And define one authoritative method such as:

step_simulation(ticks: int = 1)

or advance_seconds(seconds: float)

Then make both the visual game and the headless runner call the same simulation functions.

The command surface for the LLM should be small and explicit. Do not expose raw scene-tree poking. Give it a JSON command API such as:

reset_world(seed)

place_building(type, x, z)

step_ticks(n)

get_world_state()

get_heroes()

get_buildings()

run_until(event, max_ticks)

select_entity(id)

set_timescale(value) for visual debugging only

Godot already has built-in networking primitives for this. For a local harness, I’d use either a tiny WebSocket server or a plain TCP server inside Godot. Godot documents both. WebSocket is usually nicer because your Python LLM harness can talk JSON to it very easily. 

My preferred setup would be:

Godot headless process

starts CommandServer.gd

listens on localhost only

Python harness launches Godot

Python sends JSON commands

Godot returns compact JSON snapshots

That keeps the LLM outside the game process, which is cleaner and safer.
