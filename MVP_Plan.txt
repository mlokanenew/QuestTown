We are building the first basic playable version of a Godot 4 game.

Game concept:
A small fantasy town attracts autonomous adventurers. The player does NOT directly control the adventurers. Instead, the player develops a few key town buildings. Those buildings influence what quests are available, what services adventurers can use, and how the town grows.

For this first playable, keep everything extremely small and simple.

==================================================
DESIGN GOAL
==================================================

Build the smallest playable loop with these steps:

1. Player places and upgrades a few town buildings
2. Adventurers arrive at the town
3. Quests are generated based on the town state
4. Adventurers choose quests automatically
5. Adventurers leave town and go on quests autonomously
6. Adventurers return with money, experience, and sometimes items/injuries
7. Adventurers spend their money in the town
8. The town earns money ONLY from adventurers staying there and buying goods/services
9. Player uses town income to improve buildings
10. Improved buildings unlock better quests and better adventurer support

This should be a very simple playable, not a deep sim.

==================================================
STRICT MVP CONSTRAINTS
==================================================

For this version:

- Only 3 buildings total
- Very small upgrade/build paths for each building
- No social identity system
- No factions
- No politics
- No direct control of adventurers
- No tactical combat
- No complicated inventory system
- No complicated town staffing system
- No big world map UI
- No large tech tree
- No more than a small number of adventurers active at once

This version should feel like:
"a tiny autonomous fantasy town that starts producing quests and adventurer traffic"

==================================================
CORE LOOP
==================================================

The core loop must be:

- Build / upgrade town services
- Attract adventurers
- Generate quests
- Adventurers pick quests on their own
- Adventurers leave and resolve quests off-screen / in simulation
- Adventurers return
- Adventurers spend money in town
- Town earns income
- Player upgrades buildings
- Better buildings unlock better quest generation and better adventurer outcomes

Important:
The town should NOT generate money passively by itself.
The town earns money only when adventurers:
- rent rooms
- buy gear
- buy healing/services

==================================================
FIRST PLAYABLE BUILDINGS
==================================================

Only implement these 3 buildings:

1. Tavern
2. Weapon Shop
3. Temple

These are the only player buildings in the first playable.

--------------------------------------------------
1. Tavern
--------------------------------------------------

Purpose:
- Attract adventurers
- Give them somewhere to stay
- Generate / reveal basic quest opportunities

Base function:
- Adventurers can arrive only if there is a Tavern
- Adventurers can pay to stay there
- Tavern contributes to quest generation

Very small build path:
- Level 1: Common Room
- Level 2: Guest Rooms
- Level 3: Notice Board

Effects by level:
- Common Room:
  - enables adventurer arrival
  - allows adventurers to stay and spend small money
- Guest Rooms:
  - increases max adventurers supported slightly
  - increases lodging income
- Notice Board:
  - improves quest generation frequency/quality slightly

--------------------------------------------------
2. Weapon Shop
--------------------------------------------------

Purpose:
- Sell equipment/services to adventurers
- Improve adventurer quest readiness
- Unlock slightly harder martial quests

Base function:
- Adventurers can spend money here before quests / after returning
- Weapon Shop affects quest success slightly

Very small build path:
- Level 1: Basic Forge
- Level 2: Better Blades
- Level 3: Armour Rack

Effects by level:
- Basic Forge:
  - allows adventurers to spend on weapons
  - provides small quest success bonus
- Better Blades:
  - increases shop income
  - unlocks slightly harder combat-oriented quests
- Armour Rack:
  - increases success/survival slightly more
  - increases adventurer equipment spending

--------------------------------------------------
3. Temple
--------------------------------------------------

Purpose:
- Provide healing and blessings
- Help adventurers recover
- Unlock undead/relic/pilgrimage style quests later in the small quest pool

Base function:
- Adventurers can spend money on healing/blessing
- Temple reduces time injured adventurers stay unavailable

Very small build path:
- Level 1: Shrine
- Level 2: Healer's Room
- Level 3: Blessed Altar

Effects by level:
- Shrine:
  - enables healing service spending
  - small injury recovery improvement
- Healer's Room:
  - improves recovery speed
  - increases temple spending
- Blessed Altar:
  - adds small survival bonus
  - unlocks slightly more dangerous spiritual/undead quests

==================================================
MAXIMUM ADVENTURERS
==================================================

Keep the number of adventurers very small.

Initial target:
- max active adventurers in town/world = 4

Possible expansion rule:
- Tavern upgrades may increase cap slightly, but keep hard cap low for now
- absolute cap for first playable = 6

Reason:
- easier to simulate
- easier to make each adventurer feel distinct
- easier to debug

==================================================
ADVENTURERS
==================================================

Adventurers are autonomous.

They should have only a few properties at first:
- id
- name
- class/career
- level
- experience
- gold
- status
- health state

Suggested starting careers:
- Mercenary
- Hunter
- Thief
- Initiate

Statuses:
- InTown
- OnQuest
- Recovering

Simple rules:
- Adventurers arrive automatically if Tavern exists and capacity allows
- Adventurers stay in town for some time
- Adventurers spend money if useful services are available
- Adventurers choose from available quests automatically
- Adventurers return after quest duration
- Adventurers may return healthy, injured, richer, dead is optional for later but not required in first basic playable

For the very first playable:
- avoid permanent death if it complicates the loop too much
- injuries are enough

==================================================
QUEST SYSTEM
==================================================

Quests must be generated automatically by the town state.

The player does NOT assign adventurers manually in this first basic playable.
Instead:
- the town generates available quests
- eligible adventurers choose quests autonomously

Keep the quest system very small.

Quest properties:
- id
- name
- type
- difficulty
- duration
- gold_reward
- xp_reward
- risk_level

Small starter quest pool:
- Clear Wolves from the Road
- Escort Merchant Cart
- Hunt Forest Beast
- Investigate Old Shrine
- Drive Off Bandits

Quest availability rules:
- Tavern level influences general quest generation
- Weapon Shop level influences martial quest availability
- Temple level influences shrine/undead/spiritual quest availability

Examples:
- Tavern L1: only simple local quests
- Tavern L3 Notice Board: more quests appear and slightly better rewards
- Weapon Shop L2+: more combat quests appear
- Temple L3+: spiritual/relic/undead quests can appear

Keep this lightweight:
- no huge quest chains
- no world map
- no branching quest decisions yet

==================================================
QUEST AUTONOMY RULES
==================================================

Adventurers should choose quests by simple AI logic.

Suggested simple decision factors:
- prefer quests within ability range
- prefer quests with better reward
- avoid quests if recovering
- maybe class/career slightly biases choices

Example:
- Mercenary prefers combat quests
- Hunter prefers beast quests
- Initiate slightly prefers shrine/spiritual quests
- Thief slightly prefers risky/high-reward quests

Keep this very simple and readable.

==================================================
QUEST RESOLUTION
==================================================

Resolve quests in simulation, not combat.

Inputs:
- adventurer class
- level
- building support bonuses
- quest difficulty
- random roll

Outputs:
- gold gained
- xp gained
- injury chance
- success/failure

On return:
- adventurer status changes back to InTown or Recovering
- adventurer gold increases
- adventurer XP increases
- adventurer may level up

==================================================
TOWN ECONOMY
==================================================

Critical rule:
The town ONLY earns money when adventurers spend money in the town.

Do NOT add passive tax income, farms, citizen production, etc.

Town income sources for first playable:
- Tavern lodging
- Weapon Shop purchases
- Temple healing/blessings

Simple spending behaviour:
- before quest: adventurer may buy gear/blessing
- after quest: adventurer may pay for lodging, gear upgrades, healing
- adventurers spending transfers money from adventurer gold to town gold

This is the heart of the first playable economy.

==================================================
PLAYER DECISIONS
==================================================

For this first playable, the player's decisions are only:

1. Which of the 3 buildings to place first/next
2. Which building to upgrade
3. In what order to improve the buildings

That is enough for now.

No policies.
No social identity.
No direct quest assignment.
No direct hero control.

The gameplay question should be:
"How do I build the town so autonomous adventurers can thrive here and spend more money here?"

==================================================
DATA INTEGRATION GOAL
==================================================

The goal is now to move from:
- "small manually curated MVP tables"

to:
- "full data-driven QuestTown tables derived from `godot_data` with a stable transformation layer"

Do NOT load raw WFRP OCR tables directly into runtime gameplay without a transformation layer.

The correct approach is:
1. keep `godot_data/wfrp_db` as source/reference input
2. add transformation scripts or importer code that convert source tables into QuestTown runtime tables
3. keep runtime gameplay data in `data/`
4. make systems consume normalized QuestTown runtime tables only

==================================================
CANONICAL DATA LAYERS
==================================================

Use three clear layers:

1. Source layer
- `godot_data/wfrp_db/*.json`
- preserves imported data and OCR review status
- never hand-edit casually during gameplay work

2. Transform layer
- scripts/tools that normalize, map, filter, and enrich source records
- responsible for generating QuestTown-specific runtime tables

3. Runtime layer
- `data/*.json`
- small, stable, simulation-friendly formats used directly by Godot systems

This separation is important so we can re-import or improve the WFRP source tables later without breaking game code.

==================================================
QUESTTOWN MAPPING RULES
==================================================

The imported WFRP data is richer than the MVP.
We need a QuestTown mapping layer that compresses it into simple sim-friendly concepts.

Career mapping rules:
- every imported career gets a stable runtime `id`
- every career maps to one simple gameplay archetype
  - examples: `martial`, `scout`, `rogue`, `faith`, `survivor`, `warden`, `runner`, `commoner`
- every career maps to one simple `quest_bias`
  - examples: `combat`, `beast`, `escort`, `spiritual`, `risky`, `local`, `support`
- every career maps to one simple `service_bias`
  - examples: `tavern`, `weapons_shop`, `temple`
- profile-advance fields remain data, but MVP systems should only consume lightweight summaries at first

Skill mapping rules:
- keep imported skill IDs stable where possible
- normalize descriptions and characteristic links
- support later hero display and simple quest scoring

Characteristic mapping rules:
- characteristics should become the source for stat naming and future stat generation
- do not immediately replace current compact stat block unless the migration remains simple
- phase in gradually so the current playable loop is not destabilized

Equipment mapping rules:
- split source equipment into:
  - items heroes can notionally own or buy
  - services buildings can sell
  - optional rules text/reference entries
- for MVP gameplay, services matter first:
  - tavern lodging/food
  - weapon shop gear purchase
  - temple healing/blessing
- full item ownership can be lightweight at first:
  - optional tags and quality tiers instead of deep inventories

OCR handling rules:
- preserve `ocr_review_required`
- do not block import of the full table if a record is mildly dirty
- mark risky records so UI/debug tools can surface them later
- only block records that are structurally unusable
